# 实施路线图（v0.1）

## 1. 文档目的

本文件用于将忆刻 v0.1 的产品目标拆解为清晰的开发阶段、阶段产出物与完成定义，帮助项目在初始化阶段按顺序推进，避免范围蔓延与返工。

---

## 2. 总体策略

忆刻 v0.1 建议采用“**先打通最小闭环，再补周边能力**”的实施策略。

优先顺序应为：

1. 工程基础与分层骨架
2. 内容管理闭环
3. 今日复习闭环
4. 提醒能力
5. 备份恢复
6. 收尾测试与发布准备

原因：

- 没有内容管理，就没有可复习数据
- 没有复习闭环，产品核心价值尚未成立
- 提醒与备份都重要，但都依赖稳定的数据基础

---

## 3. 阶段划分总览

| 阶段 | 名称 | 目标 |
|---|---|---|
| P0 | 工程基础 | 建立分层、导航、基础依赖与目录结构 |
| P1 | 内容管理闭环 | 实现 Deck / Card / Question 的增删改查 |
| P2 | 今日复习闭环 | 实现首页概览、复习流程、评分调度 |
| P3 | 提醒能力 | 实现固定时间通知提醒 |
| P4 | 备份恢复 | 实现 JSON 导出与全量恢复 |
| P5 | 稳定化与验收 | 做测试、修边界、准备首版可用状态 |

---

## 4. P0：工程基础

### 4.1 目标

在不做复杂业务前，先把项目结构搭好。

### 4.2 建议任务

- 建立 `ui / domain / data` 分层包结构
- 引入 Navigation Compose
- 确定应用主题、基础颜色、基础组件风格
- 建立首页、卡组、设置三个基础路由
- 预留 `ViewModel` 与 `UseCase` 目录
- 引入 Room、DataStore、WorkManager 依赖
- 建立统一时间工具与结果类型封装

### 4.3 产出物

- 可运行的导航骨架
- 可编译的分层目录
- 基础主题与页面占位
- 数据层技术栈已接入

### 4.4 完成定义

- 页面已能正常导航
- Room/DataStore/WorkManager 依赖可用
- 项目目录结构与命名约定已经固定

---

## 5. P1：内容管理闭环

### 5.1 目标

让用户能真正录入和维护自己的复习内容。

### 5.2 建议任务

- 实现 `Deck`、`Card`、`Question` 的 Room 实体和 DAO
- 实现卡组列表页
- 实现卡片列表页
- 实现问题编辑页
- 实现新建、编辑、删除、归档逻辑
- 保存后正确生成新问题初始调度字段

### 5.3 产出物

- 可创建卡组
- 可创建卡片
- 可在卡片下录入多个问题
- 可编辑与删除内容

### 5.4 完成定义

- 从 UI 到数据库的数据流打通
- 退出应用后内容仍能正确保留
- 新建问题后具备合法的初始 `stageIndex` 与 `dueAt`

---

## 6. P2：今日复习闭环

### 6.1 目标

打通产品最核心的“看到问题 -> 显示答案 -> 评分 -> 重新调度”闭环。

### 6.2 建议任务

- 实现首页今日待复习统计
- 实现到期问题查询
- 实现复习队列
- 实现复习页
- 实现四档评分
- 实现调度器
- 写入 `ReviewRecord`
- 本卡完成后切换下一卡或结束

### 6.3 产出物

- 首页可显示今日待复习数量
- 用户可开始复习
- 评分后数据库状态正确更新
- 同一天不会重复刷回已评分问题

### 6.4 完成定义

- 核心复习流程可连续完成
- 调度结果与文档规则一致
- 中途退出后已评分进度不会丢失

---

## 7. P3：提醒能力

### 7.1 目标

让用户每天在固定时间收到复习提醒。

### 7.2 建议任务

- 设置页支持提醒开关
- 设置页支持时间选择
- DataStore 保存提醒配置
- 接入 WorkManager
- 实现通知渠道与通知展示
- 到点时检查是否有待复习内容
- 设备重启后重建提醒

### 7.3 产出物

- 可开启提醒
- 可修改提醒时间
- 有待复习内容时能收到通知

### 7.4 完成定义

- 提醒设置持久化成功
- 通知触发逻辑与数据状态一致
- 权限拒绝时有明确提示

---

## 8. P4：备份恢复

### 8.1 目标

确保本地数据可导出、可恢复，不依赖网络。

### 8.2 建议任务

- 设计备份 JSON 序列化结构
- 实现导出功能
- 实现系统文件选择器接入
- 实现导入、校验、全量恢复
- 恢复后重建提醒
- 处理错误文件与失败回滚

### 8.3 产出物

- 可导出完整 JSON 备份
- 可从合法 JSON 恢复全部数据
- 恢复失败不破坏现有数据

### 8.4 完成定义

- 导入导出均可稳定运行
- 恢复后层级关系、due 数据、复习历史正确
- 风险提示与确认流程清晰

---

## 9. P5：稳定化与验收

### 9.1 目标

把“能跑”提升到“可长期自用”。

### 9.2 建议任务

- 补齐调度单元测试
- 补齐 DAO 测试
- 补齐关键 UI 测试
- 统一空状态和错误文案
- 修复边界问题
- 检查版本号与备份版本
- 做一次完整手动回归

### 9.3 完成定义

- 核心闭环场景全部通过
- 已知高风险问题清零或明确记录
- 项目进入可持续开发状态

---

## 10. 推荐实施顺序中的依赖关系

### 10.1 必须先完成

- `P0` 先于所有阶段
- `P1` 先于 `P2`
- `P2` 先于 `P3` 和 `P4`

### 10.2 可部分并行

- `P3` 与 `P4` 在 `P2` 稳定后可并行推进
- `P5` 从 `P2` 开始即可逐步插入，不必完全压后

---

## 11. 每阶段建议交付粒度

不要按“页面”切分过细，而应按“可用能力”切分。

例如：

- 一次交付“卡组与卡片管理可用”
- 一次交付“复习评分与调度可用”
- 一次交付“备份恢复可用”

不建议只交付：

- 一个纯静态页面
- 只有 UI 没有数据流的壳子

---

## 12. 风险与缓冲建议

### 12.1 高风险点

- 调度规则实现与文档不一致
- 复习流程状态切换混乱
- 恢复失败污染原数据
- 提醒逻辑在系统限制下不稳定

### 12.2 缓冲策略

- 调度器先写单元测试再接 UI
- 恢复功能必须走事务
- 复习页先做最小闭环，不加动效
- 通知先保证“可用”，不追求“绝对准时”

---

## 13. 最小可用版本判断

当满足以下条件时，可视为 v0.1 已达到最小可用状态：

1. 用户能录入和编辑内容
2. 首页能显示今日待复习
3. 用户能完成复习评分流程
4. 系统能更新下次复习时间
5. 提醒可正常工作
6. 数据可备份与恢复

---

## 14. 结论

忆刻 v0.1 的开发顺序不应是“先把页面铺满”，而应是：

- 先搭骨架
- 再打通内容录入
- 再完成复习闭环
- 最后补提醒与备份

只要严格按这个顺序推进，返工成本会明显更低。
